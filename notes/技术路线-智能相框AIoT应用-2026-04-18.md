# 智能相框 AIoT 应用技术路线

**文档版本**：2026-04-18  
**适用项目**：`SmartFrameCloud`  
**定位**：基于 Web + PHP + MySQL + MQTT 的智能相册/智能相框云平台前后端一体化实现说明

## 1. 项目概述

### 1.1 项目目标

本项目用于构建一个面向智能相框设备的云端管理平台，提供以下核心能力：

- 用户注册、登录、验证码校验
- 相册创建、浏览、封面设置
- 图片上传、存储、展示
- 图片转换为设备可显示的 BMP 格式
- 智能相框设备绑定、解绑、列表管理
- 通过 MQTT 将图片下发到指定设备

### 1.2 当前形态

当前代码库是一个轻量级单体 Web 应用，前端页面与后端接口部署在同一站点目录中：

- 前端：原生 HTML + CSS + JavaScript
- 后端：PHP 8.1，按接口目录组织
- 数据库：MySQL
- 设备通信：MQTT（EMQX Broker）
- 文件存储：本地磁盘 `uploads/` 与 `cache/`

该形态适合中小规模验证、设备联调、功能迭代和部署上线前的快速闭环。

## 2. 总体技术路线

### 2.1 架构思路

项目采用“浏览器管理端 + PHP API + MySQL + MQTT + 本地文件存储”的经典 AIoT 控制平台路线：

1. 用户通过浏览器访问云平台页面。
2. 前端使用 `fetch` 请求 PHP API，完成用户认证、相册管理、图片上传、设备管理等操作。
3. 后端将业务数据写入 MySQL，将原图保存到 `uploads/`。
4. 当用户下发图片时，后端将原图转换为适配电子纸/彩屏相框的 BMP 文件，并缓存到 `cache/`。
5. 后端构造 MQTT 消息，将 BMP 下载地址推送到指定设备主题。
6. 设备订阅对应主题后获取图片地址，再通过 HTTP 下载 BMP 文件进行显示。

### 2.2 系统组成

```text
浏览器管理端
  ├─ 登录 / 注册
  ├─ 控制台
  ├─ 相册管理
  └─ 设备管理
        |
        v
PHP 后端 API
  ├─ auth
  ├─ user
  ├─ albums
  ├─ upload
  ├─ imgprocess
  └─ devices
        |
        +--> MySQL（用户、相册、图片、设备、下发日志）
        +--> uploads/（原图）
        +--> cache/（BMP 缓存）
        |
        v
MQTT Broker（EMQX）
        |
        v
智能相框设备
```

### 2.3 技术选型

#### 前端

- HTML5 多页面结构
- CSS3 统一样式文件 `css/style.css`
- 原生 JavaScript
- `fetch` + `FormData` 调用后端接口
- 模态框交互、图片预览、相册/设备列表动态渲染

#### 后端

- PHP 8.1
- PDO 访问 MySQL
- Session 维持登录态
- GD 库生成验证码、处理图片和生成 BMP
- 自实现 MQTT v3.1.1 报文发送逻辑

#### 数据与存储

- MySQL：保存用户、相册、图片、设备、图片下发记录
- `uploads/`：按年月分目录保存原始图片
- `cache/`：按原图相对路径生成并缓存 BMP 文件

## 3. 前端实现路线

### 3.1 页面结构

当前项目前端由 5 个核心页面组成：

- `index.html`：登录页
- `register.html`：注册页
- `dashboard.html`：控制台
- `albums.html`：相册管理页
- `devices.html`：设备管理页

### 3.2 交互模式

前端采用“页面渲染 + 接口驱动”的实现方式：

- 登录/注册通过表单提交到认证接口
- 控制台读取用户信息与统计数据
- 相册页可创建相册、查看相册图片、上传图片、设置封面
- 设备页可绑定设备、解绑设备、选择图片下发
- 所有业务接口均返回统一 JSON 结构：`success`、`message`、`data`

### 3.3 前端特点

- 无前端框架依赖，部署简单
- 样式统一在单文件中维护，适合快速迭代
- 相册、设备、图片列表均通过接口动态加载
- 交互逻辑清晰，便于后续迁移到 Vue/React 等框架

## 4. 后端模块设计

### 4.1 公共基础层

#### 配置文件

`backend/config/database.php` 统一维护：

- MySQL 连接信息
- MQTT 连接信息
- 站点地址 `APP_URL`
- 上传路径 `UPLOAD_PATH`
- 上传限制 `MAX_UPLOAD_SIZE`
- Session 初始化
- 默认时区 `Asia/Shanghai`

#### 通用函数

`backend/includes/functions.php` 负责：

- JSON 响应输出
- 登录鉴权
- 验证码生成与校验
- 设备 UID 校验
- 图片上传与数据库写入
- MQTT 报文编码与发布
- 图片下发设备的完整业务流程

### 4.2 认证模块

接口目录：`backend/api/auth/`

已实现接口：

- `captcha.php`：生成验证码图片
- `login.php`：账号登录，支持用户名或邮箱
- `register.php`：用户注册并自动创建默认相册
- `logout.php`：退出登录

认证路线：

1. 用户访问登录/注册页，先获取验证码。
2. 后端通过 Session 保存验证码与有效期。
3. 登录时校验用户名/邮箱、密码、验证码。
4. 登录成功后将 `user_id` 写入 Session。
5. 受保护接口统一通过 `requireAuth()` 校验登录状态。

### 4.3 用户信息模块

接口目录：`backend/api/user/`

已实现接口：

- `profile.php`

返回内容：

- 当前用户基本信息
- 相册数量
- 图片数量
- 设备数量

该接口为控制台首页提供统计数据来源。

### 4.4 相册模块

接口目录：`backend/api/albums/`

已实现接口：

- `list.php`：获取相册列表及图片数量
- `create.php`：创建相册
- `delete.php`：删除相册及其物理图片文件
- `images.php`：获取指定相册下的图片列表
- `set-cover.php`：设置相册封面

实现特征：

- 相册只可访问当前登录用户自己的数据
- 相册支持封面图字段 `cover_image_id`
- 上传首张图片时会自动设为封面
- 删除相册时同步删除 `uploads/` 中的原图文件

### 4.5 图片上传模块

接口目录：`backend/api/upload/`

已实现接口：

- `upload.php`

上传路线：

1. 校验用户登录状态与相册归属。
2. 校验文件上传状态、大小、图片格式。
3. 使用 `getimagesize()` 获取图片尺寸与 MIME 类型。
4. 生成唯一文件名。
5. 按 `Y/m` 创建目录并保存到 `uploads/`。
6. 写入 `images` 表。
7. 若相册无封面，则自动设当前图片为封面。

当前允许格式：

- JPEG
- PNG
- GIF
- WebP

当前大小限制：

- 10 MB

说明：

- 上传链路允许 GIF、WebP
- 但当前 BMP 生成核心 `generate_bmp()` 实际仅处理 JPEG、PNG
- 因此 GIF、WebP 图片虽然可以入库上传，但未必能成功进入设备下发链路

### 4.6 图片处理与 BMP 生成模块

接口目录：`backend/api/imgprocess/`

已实现接口：

- `imgprocess.php`

功能说明：

- 支持通过 `image_id`、上传相对路径 `file`、绝对路径 `source` 三种方式触发处理
- 将图片统一转换为 `800 x 480`
- 使用 7 色调色板进行有序抖动
- 输出标准 24 位 BMP V3 文件
- 生成结果缓存到 `cache/年/月/文件名.bmp`

图像处理路线：

1. 读取原始图片
2. 等比缩放并居中裁剪至设备目标分辨率
3. 应用 Bayer 矩阵有序抖动
4. 映射到 7 色调色板
5. 写出 BMP 文件
6. 命中缓存时直接复用，避免重复计算

当前调色板颜色：

- 黑
- 白
- 红
- 黄
- 蓝
- 绿
- 橙

### 4.7 设备管理模块

接口目录：`backend/api/devices/`

已实现接口：

- `list.php`：获取设备列表
- `bind.php`：绑定设备
- `unbind.php`：解绑设备
- `send-image.php`：将图片下发到指定设备

设备管理规则：

- 设备唯一标识为 16 位十六进制字符串，表示 64 位芯片 ID
- 绑定时如果设备已存在且未被占用，则补充绑定信息
- 若设备不存在，则自动创建并绑定
- 解绑时清空用户归属、设备名称、描述、当前图片等字段

当前设备状态定义：

- `0`：未绑定
- `1`：在线
- `2`：离线

注意：

- 当前代码在绑定设备后默认写入 `status = 2`
- 项目尚未实现设备心跳/在线状态回传，所以设备一般会持续显示“离线”

### 4.8 设备图片下发模块

图片下发核心逻辑位于 `backend/includes/functions.php` 中的 `sendImageToDevice()`。

下发路线：

1. 校验设备归属与图片归属。
2. 找到原图路径。
3. 调用 `process_image_to_bmp()` 生成 BMP。
4. 拼接 BMP 下载地址，当前使用 `imget.php?file=...` 对外提供访问。
5. 以 MQTT 主题 `device/{device_uid}/image` 发布 JSON 消息。
6. 将下发记录写入 `device_image_logs`。
7. 若发布成功，更新设备当前图片与日志状态；失败则写入错误信息。

MQTT 消息结构包含：

- `image_id`
- `image_url`
- `original_name`
- `width`
- `height`
- `timestamp`

### 4.9 BMP 文件访问模块

入口文件：`imget.php`

职责：

- 从 `cache/` 中读取 BMP 文件
- 限制只能访问 `.bmp`
- 进行路径清洗，避免目录遍历
- 设置 `Content-Type: image/bmp`
- 用于设备通过 HTTP 下载最终显示资源

## 7. 部署路线

### 7.1 运行环境

从现有代码与部署文档可确认推荐环境为：

- Nginx
- PHP 8.1
- MySQL
- EMQX
- PHP GD 扩展

### 7.2 关键配置项

`backend/config/database.php` 中包含以下关键配置：

- 数据库主机、库名、账号、密码
- MQTT 主机、端口、账号、密码
- 站点访问地址 `APP_URL`
- 上传目录 `UPLOAD_PATH`
- 上传访问前缀 `UPLOAD_URL`

### 7.3 设备资源访问方式

项目当前采用以下方式给设备提供图片：

- 原图访问：`/uploads/...`
- BMP 访问：`imget.php?file=年/月/文件.bmp`

这意味着设备只需要具备：

- MQTT 订阅能力
- HTTP 下载 BMP 文件能力

## 8. 已实现能力与当前边界

### 8.1 已实现能力

- 账号注册、登录、退出、验证码
- 用户统计信息读取
- 相册创建、删除、浏览
- 图片上传
- 相册封面设置
- 设备绑定、解绑、列表展示
- 图片转 BMP 并缓存
- MQTT 下发图片到设备
- BMP 文件安全读取

### 8.2 当前未闭环或待完善部分

结合代码现状与 `notes/0-todo-list.md`，当前仍有以下边界：

- 图片删除按钮已在前端预留，但后端删除图片接口尚未实现
- 设备在线状态监测尚未实现，缺少心跳 Topic 与状态回写逻辑
- 相册页尚未直接提供“从相册下发到设备”的联动入口
- 图片上传与图片转 BMP 的格式支持不完全一致，GIF/WebP 目前可能无法完成最终下发
- 设备下发使用的 BMP 访问地址仍存在固定公网地址写法，部署时需要特别校对
- 前端仍是多页面原生实现，后续如功能增加，状态管理复杂度会上升
- 配置文件中仍存在部署环境硬编码项，生产环境需要进一步参数化

## 9. 下一阶段技术演进建议

### 9.1 设备侧联通性增强

建议新增设备状态上报协议，例如：

- `device/{device_uid}/msg`
- `device/{device_uid}/status`

后端收到设备在线心跳后，可将 `devices.status` 更新为在线，并记录最后活跃时间。

### 9.2 图片管理增强

建议补齐以下能力：

- 单张图片删除
- 批量上传与批量删除
- 图片排序
- 从相册直接下发到设备

### 9.3 架构演进方向

当设备量和用户量上升后，可考虑：

- 将 MQTT 通信从 PHP 页面请求中拆出为独立服务
- 引入对象存储替代本地磁盘
- 为图片处理建立异步任务队列
- 将前端迁移到组件化框架

## 10. 当前结论

`SmartFrameCloud` 当前已经完成了智能相框云平台的主干闭环：

- 管理端可完成用户、相册、图片、设备的基础管理
- 后端可将原图转换为设备适配 BMP
- 平台可通过 MQTT 向设备下发图片资源地址
- 设备可通过 HTTP 获取最终 BMP 文件显示

从技术路线角度看，本项目已经具备一个可运行 AIoT 智能相框云平台的核心雏形，下一阶段重点应放在设备在线状态闭环、图片管理补全、配置规范化和可扩展部署能力上。
