# 后端调试接口 GET/POST

**文档日期**：2026-04-19  
**适用项目**：`SmartFrameCloud`  
**目标读者**：不懂 PHP，但需要自己验证前后端配合是否正常的人

## 1. 先说最重要的

当前项目里，和“下发图片 / 生成 BMP / 联调设备”最相关的接口有 4 个：

1. `backend/api/devices/debug-render.php`
作用：只调试 BMP 生成，不碰设备、不发 MQTT。

2. `backend/api/devices/debug-send-image.php`
作用：调试“准备下发”的全过程。默认是 `dry-run`，不真正发 MQTT；只有 `publish=1` 才会真的发。

3. `backend/api/devices/send-image.php`
作用：正式下发接口，前端页面点“下发到设备”时走的就是它。只支持 `POST`。

4. `backend/api/imgprocess/imgprocess.php`
作用：底层图片转 BMP 接口，可单独测试图片处理链路。支持 `GET / POST`。

如果你只是想安全验证前后端联调，推荐顺序是：

1. 先测 `debug-render.php`
2. 再测 `debug-send-image.php` 的默认 `dry-run`
3. 最后确认没问题了，再测 `send-image.php` 或 `debug-send-image.php?publish=1`

## 2. 调试前准备

## 2.1 必须先登录

这些接口大多都要求登录状态。  
最简单的做法是：

1. 先在浏览器里正常登录云平台
2. 保持这个浏览器标签页不要退出
3. 再直接在同一个浏览器地址栏里访问下面的 GET 调试地址

如果你用 Postman / Apifox 发请求，也需要带上登录后的 Cookie；不带的话通常会返回：

```json
{"success":false,"message":"请先登录","data":[]}
```

## 2.2 你要先知道两个 ID

调试时最常用的是：

- `image_id`
- `device_id`

### 获取 `device_id`

登录后，在浏览器打开：

```text
http://47.108.232.40:2026/cloud/backend/api/devices/list.php
```

返回 JSON 里的每个设备对象，都会有：

- `id`
- `device_uid`
- `name`

这里的 `id` 就是 `device_id`。

### 获取 `image_id`

先打开相册列表：

```text
http://47.108.232.40:2026/cloud/backend/api/albums/list.php
```

拿到某个相册的 `id` 之后，再打开：

```text
http://47.108.232.40:2026/cloud/backend/api/albums/images.php?album_id=相册ID
```

返回 JSON 里的每张图片都有：

- `id`
- `original_name`
- `file_path`

这里的 `id` 就是 `image_id`。

## 3. 统一参数说明

这几个接口目前共用的图片调整参数基本一致：

- `orientation`
可选值：`landscape` / `portrait`

- `rotate`
可选值：`0` / `90` / `180` / `270`

- `crop_center_x`
范围：`0 ~ 1`
含义：裁剪中心点横向位置，`0.5` 是正中间

- `crop_center_y`
范围：`0 ~ 1`
含义：裁剪中心点纵向位置，`0.5` 是正中间

- `crop_zoom`
范围：`0.5 ~ 4`
含义：
- `1` 是默认
- 小于 `1` 是缩小视图
- 大于 `1` 是放大裁剪

当前最新逻辑：

- 当 `crop_zoom=0.5` 这类缩小时，图片可能无法铺满整个画布
- 这时空白区域会补成白色
- 不是透明
- 因为透明在墨水屏上容易显示成黑色，效果很差

## 4. 推荐先测：`debug-render.php`

## 4.1 作用

只验证：

- 图片能不能找到
- 参数有没有被正确归一化
- BMP 能不能成功生成
- 生成后的 BMP 地址是什么

它不会下发设备，不会发 MQTT，最适合先排查图片处理问题。

## 4.2 GET 调试示例

把下面地址直接粘到浏览器地址栏：

```text
http://47.108.232.40:2026/cloud/backend/api/devices/debug-render.php?image_id=123&orientation=landscape&rotate=90&crop_center_x=0.5&crop_center_y=0.5&crop_zoom=1
```

再比如测试“缩小到 0.5 倍并留白”：

```text
http://47.108.232.40:2026/cloud/backend/api/devices/debug-render.php?image_id=123&orientation=landscape&rotate=0&crop_center_x=0.5&crop_center_y=0.5&crop_zoom=0.5
```

## 4.3 POST 调试示例

如果你用 Postman / Apifox：

- 方法：`POST`
- 地址：

```text
http://47.108.232.40:2026/cloud/backend/api/devices/debug-render.php
```

- Body 选 `x-www-form-urlencoded` 或 `form-data`
- 填这些字段：

```text
image_id=123
orientation=landscape
rotate=90
crop_center_x=0.5
crop_center_y=0.5
crop_zoom=1
```

## 4.4 看返回结果时重点看什么

成功时主要看：

- `success=true`
- `data.source_exists=true`
- `data.source_readable=true`
- `data.render_options`
- `data.bmp_result`
- `data.bmp_url`

你可以把 `bmp_url` 单独复制到浏览器里打开，看 BMP 最终生成效果。

## 5. 最实用联调：`debug-send-image.php`

## 5.1 作用

这个接口是最适合联调“前后端 + 设备下发逻辑”的接口。

默认行为是：

- 只做 `dry-run`
- 不写正式下发日志
- 不更新设备当前图片
- 不真的发 MQTT

所以它非常适合反复试。

## 5.2 GET 调试示例

```text
http://47.108.232.40:2026/cloud/backend/api/devices/debug-send-image.php?device_id=1&image_id=123&orientation=landscape&rotate=180&crop_center_x=0.5&crop_center_y=0.5&crop_zoom=0.5
```

这条请求会返回：

- 当前设备信息
- 当前图片信息
- 源文件路径
- BMP 生成结果
- 准备发给设备的 MQTT `topic`
- 准备发给设备的 MQTT `payload`
- 当前是否 `publish`

## 5.3 真发 MQTT 的方式

如果你已经确认 dry-run 返回都正常，再用：

```text
http://47.108.232.40:2026/cloud/backend/api/devices/debug-send-image.php?device_id=1&image_id=123&orientation=landscape&rotate=90&crop_center_x=0.5&crop_center_y=0.5&crop_zoom=1&publish=1
```

注意：

- `publish=1` 才会真的发 MQTT
- 这个接口即便真发，也主要用于调试
- 正式业务还是建议走前端页面或正式接口 `send-image.php`

## 5.4 POST 调试示例

- 方法：`POST`
- 地址：

```text
http://47.108.232.40:2026/cloud/backend/api/devices/debug-send-image.php
```

- Body 示例：

```text
device_id=1
image_id=123
orientation=portrait
rotate=270
crop_center_x=0.5
crop_center_y=0.5
crop_zoom=0.8
publish=0
```

## 5.5 结果怎么看

成功时重点看：

- `data.bmp_result.success`
- `data.topic`
- `data.payload.image_url`
- `data.payload.width`
- `data.payload.height`
- `data.payload.orientation`
- `data.payload.rotate`
- `data.mqtt_published`

经验上：

- 如果 `bmp_result.success=true`，但设备没反应，优先看 MQTT
- 如果 `bmp_result.success=false`，优先看图片路径、渲染参数、GD 库或缓存文件

## 6. 正式接口：`send-image.php`

## 6.1 作用

这是页面“下发到设备”按钮真正调用的正式接口。  
它只支持 `POST`。

它和 `debug-send-image.php` 的最大区别是：

- 会走正式下发流程
- 会更新设备当前图片
- 会写入下发日志

## 6.2 POST 示例

- 方法：`POST`
- 地址：

```text
http://47.108.232.40:2026/cloud/backend/api/devices/send-image.php
```

- Body：

```text
device_id=1
image_id=123
orientation=landscape
rotate=90
crop_center_x=0.5
crop_center_y=0.5
crop_zoom=1
```

如果只是调试，不建议一上来就测这个，建议先测 `debug-send-image.php`。

## 7. 底层接口：`imgprocess.php`

## 7.1 作用

这是最底层的图片处理接口。  
适合做下面两种事：

1. 不经过设备接口，单测 BMP 生成
2. 已知文件路径时，直接测处理是否成功

## 7.2 三种输入方式

它目前支持三种指定源图的方式：

1. `source=绝对路径`
2. `file=uploads 下的相对路径`
3. `image_id=图片ID`

一般最推荐第三种：`image_id`

## 7.3 GET 示例

```text
http://47.108.232.40:2026/cloud/backend/api/imgprocess/imgprocess.php?image_id=123&orientation=landscape&rotate=270&crop_center_x=0.5&crop_center_y=0.5&crop_zoom=0.5
```

成功后会返回：

- `bmp_file`
- `bmp_url`
- `target_width`
- `target_height`

## 8. 最简单的调试流程

如果你不懂 PHP，最推荐按下面走：

1. 登录网站，不要退出
2. 打开 `devices/list.php`，记下 `device_id`
3. 打开 `albums/list.php` 和 `albums/images.php`，记下 `image_id`
4. 用浏览器直接访问 `debug-render.php`
先确认 BMP 能生成
5. 再访问 `debug-send-image.php`
先用默认 dry-run 看返回的 `topic` 和 `payload`
6. 确认没问题后，再加 `publish=1`
7. 最后如果要模拟真实前端行为，再用 Postman/Apifox 调 `send-image.php`

## 9. 常见报错怎么理解

## 9.1 `请先登录`

原因：

- 没登录
- 或 Postman/Apifox 没带 Cookie

## 9.2 `图片不存在或无权访问`

原因：

- `image_id` 错了
- 这张图不属于当前登录用户

## 9.3 `设备不存在或未绑定到当前用户`

原因：

- `device_id` 错了
- 设备还没绑定到当前账号

## 9.4 `BMP 生成失败`

优先排查：

- `source_exists`
- `source_readable`
- `render_options`
- `bmp_result.message`

## 9.5 `MQTT 发布失败`

说明：

- 图片已经处理完了
- 但消息没有成功发到 MQTT Broker

优先排查：

- MQTT 服务是否在线
- 用户名密码是否正确
- Broker 是否允许连接

## 10. 一句话建议

想安全调试，先用：

- `debug-render.php`
- `debug-send-image.php`

想模拟真实业务，再用：

- `send-image.php`

想确认最新图片调整逻辑是否生效，重点测这三组参数：

```text
crop_zoom=0.5
rotate=180
orientation=portrait
```

因为这三组最容易暴露“缩小留白、旋转方向、横竖屏尺寸”是否和前端预览一致。
