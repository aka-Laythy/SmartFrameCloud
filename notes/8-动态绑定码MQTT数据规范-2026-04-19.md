# 动态绑定码 MQTT 数据规范

**文档日期**：2026-04-19  
**适用范围**：SmartFrameCloud 云平台向设备下发动态绑定码  
**对应服务端脚本**：`backend/mqtt/publish_bind_codes.php`  
**对应公共函数**：`backend/includes/functions.php` 中的 `publishDynamicBindCodeToDevice()`

## 1. 文档目的

本文档用于约定云平台向 ESP32-S3 设备下发“动态绑定码”时使用的 MQTT topic、payload、发布时机和设备端处理建议。

这份文档描述的是：

- **云平台下发**
- **设备接收**

不是设备上行注册协议。  
设备上行注册协议请参考：

- `notes/6-subscribe_bound.php接收的MQTT数据规范-2026-04-19.md`

## 2. 功能定位

动态绑定码的作用是：

- 让设备在墨水屏上显示一个 6 位数字
- 用户在云平台“绑定设备”弹窗里只输入这 6 位数字
- 云平台据此找到对应的未绑定设备并完成绑定

所以动态绑定码消息本质上是：

- 一条**云端到设备**的下行控制消息

## 3. Topic 规范

## 3.1 标准格式

云平台向设备下发动态绑定码时，使用如下 topic：

```text
device/<DEVICE_UID>/bound
```

例如：

```text
device/AC12EF34AB56CD78/bound
```

## 3.2 为什么仍然使用 `bound`

当前系统里：

- 设备首次注册上行：也走 `device/<DEVICE_UID>/bound`
- 云端动态绑定码下发：也走 `device/<DEVICE_UID>/bound`

这不是最理想的长期设计，但当前已经通过 payload 中的 `event` 字段做了明确区分：

- 设备上行注册：`event = reg_new_device`
- 云端下行绑定码：`event = dyn_bound_code`

因此：

- 固件端必须按 `event` 处理
- 服务端 subscriber 也必须按 `event` 区分上下行语义

## 4. Payload 规范

## 4.1 标准 JSON 结构

当前云平台下发动态绑定码的标准 payload 为：

```json
{
  "event": "dyn_bound_code",
  "device_uid": "AC12EF34AB56CD78",
  "dyn_bound_code": "483726",
  "expires_in": 300,
  "timestamp": 1776530000
}
```

## 4.2 字段说明

- `event`
  - 固定值：`dyn_bound_code`
  - 用于标识这是“动态绑定码下发消息”

- `device_uid`
  - 设备 16 位十六进制唯一 ID
  - 推荐全大写
  - 应与 topic 中的 `DEVICE_UID` 保持一致

- `dyn_bound_code`
  - 6 位数字动态绑定码
  - 例如：`483726`

- `expires_in`
  - 剩余有效秒数
  - 当前默认值：`300`
  - 即 5 分钟

- `timestamp`
  - 云平台发送该消息时的 Unix 时间戳

## 4.3 当前绑定码规则

当前后端实现中的动态绑定码规则是：

- 长度：6 位
- 字符集：仅数字 `0-9`
- 示例：
  - `483726`
  - `015902`
  - `900318`

设备端收到后应按字符串处理，不要转成整数后再显示，否则：

- `015902`

这类前导零会丢失。

## 5. 与图片下发消息的风格对齐

你之前要求“格式可以参考图片下发（`device/<DEVICE_UID>/image`）的格式”，当前动态绑定码消息已经与图片下发保持了同一风格：

### 图片下发格式

```json
{
  "image_id": 12,
  "image_url": "http://...",
  "original_name": "xxx.bmp",
  "width": 800,
  "height": 480,
  "timestamp": 1776530000
}
```

### 动态绑定码格式

```json
{
  "event": "dyn_bound_code",
  "device_uid": "AC12EF34AB56CD78",
  "dyn_bound_code": "483726",
  "expires_in": 300,
  "timestamp": 1776530000
}
```

一致点是：

- 都使用 JSON
- 都包含 `timestamp`
- 都以 topic 来确定目标设备
- payload 只携带该次业务所需字段

## 6. 发布时机

## 6.1 云平台什么时候会下发

当前后端逻辑中，动态绑定码是由：

- `backend/mqtt/publish_bind_codes.php`

批量扫描后统一下发的。

它会处理：

- `user_id IS NULL`
- `status = 1`
- 动态绑定码为空
- 或动态绑定码即将过期/已经过期

的设备。

## 6.2 当前轮转策略

当前后端实现采用：

- 绑定码有效期：5 分钟
- 提前刷新窗口：60 秒
- 单轮最大处理设备数：100 台
- 单设备发布间隔：50 ms

也就是说：

- 不是每分钟所有设备都重发
- 只有需要新码的设备才会被处理

这样可以兼顾：

- 用户体验
- MQTT 压力
- 数据库压力

## 7. MQTT 参数建议

## 7.1 QoS

当前建议云平台下发时使用：

```text
QoS 0
```

原因：

- 当前 `publishMQTT()` 实现是轻量版 PUBLISH
- 没有完整实现 QoS 1/2 的发布确认链路
- 设备端绑定码会定时轮转，短时间内即使偶发丢一条，也会在后续重新收到

如果后续要进一步提高可靠性，可以再升级发布端实现。

## 7.2 Retain

当前**不建议**对动态绑定码消息使用 retained message。

原因：

- 绑定码有短时效
- retained 容易导致设备重连后拿到旧码
- 设备墨水屏可能显示已经失效的绑定码

建议：

```text
retain = false
```

## 8. 设备端接收要求

设备端订阅相关消息时，至少应做到：

### 8.1 校验 topic

只处理：

```text
device/<本机DEVICE_UID>/bound
```

### 8.2 校验 JSON

收到消息后必须先解析 JSON。

### 8.3 校验 `event`

只有当：

```json
"event": "dyn_bound_code"
```

时，才按“显示动态绑定码”的逻辑处理。

### 8.4 校验 `device_uid`

payload 里的 `device_uid` 应与本机一致。  
如果不一致，建议忽略。

### 8.5 校验 `dyn_bound_code`

设备端建议校验：

- 必须是字符串
- 长度必须为 6
- 必须全部是数字

### 8.6 处理 `expires_in`

设备端建议：

- 收到后记录本地接收时间
- 屏幕显示“5分钟内有效”
- 若后续收到新码，则覆盖旧码显示

## 9. 设备端推荐处理逻辑

伪流程如下：

```text
收到 MQTT 消息
  -> 检查 topic 是否是 device/<本机UID>/bound
  -> 解析 JSON
  -> 检查 event 是否等于 dyn_bound_code
  -> 检查 payload.device_uid 是否等于本机UID
  -> 检查 dyn_bound_code 是否为6位数字
  -> 保存当前绑定码
  -> 墨水屏显示绑定码
  -> 显示“5分钟内有效”
```

## 10. 设备屏幕展示建议

收到动态绑定码后，建议设备墨水屏展示为：

```text
绑定设备

动态绑定码：
483726

5分钟内有效
请在云平台输入此码
```

如果设备稍后又收到新的绑定码，应覆盖旧内容显示。

## 11. 失效与覆盖规则

当前推荐规则：

- 新绑定码下发后，旧绑定码自动失效
- 用户成功绑定后，云平台会清空数据库中的 `dyn_bound_code`
- 设备之后如果还收到新绑定码，说明设备仍处于“未绑定且在线”状态

对设备端来说，最重要的处理原则是：

- **永远显示最后一次收到的合法绑定码**

## 12. 与数据库的关系

当前绑定码相关字段位于 `devices` 表：

- `dyn_bound_code`
- `dyn_bound_code_issued_at`
- `dyn_bound_code_expires_at`

云平台下发绑定码前会先写库，再发 MQTT。

因此：

- 数据库中的码是“权威状态”
- MQTT 消息是“通知设备展示”

## 13. 当前版本边界

当前这份规范也有一些边界：

- 没有消息签名
- 设备端不一定校验 `timestamp`
- MQTT 下发目前是轻量实现，不是完整企业级可靠投递
- topic 仍与“设备注册上行”共用 `bound`

所以当前协议更适合：

- 第一阶段产品联调
- 小规模真实部署

而不是：

- 大规模复杂设备管理场景

## 14. 最终推荐规范

如果你现在就要给 ESP32-S3 固件端一个可以直接照着收消息的版本，请按下面这份：

### Topic

```text
device/<16位全大写十六进制DEVICE_UID>/bound
```

### Payload

```json
{
  "event": "dyn_bound_code",
  "device_uid": "<16位全大写十六进制DEVICE_UID>",
  "dyn_bound_code": "483726",
  "expires_in": 300,
  "timestamp": 1776530000
}
```

### 设备端处理规则

- 只处理 `event = dyn_bound_code`
- `device_uid` 必须与本机一致
- `dyn_bound_code` 必须为 6 位数字
- 收到后立即更新屏幕显示

## 15. 与当前代码对应关系

对应文件：

- `backend/mqtt/publish_bind_codes.php`
- `backend/includes/functions.php`
- `backend/api/devices/bind.php`

当前核心逻辑是：

1. 云平台定时扫描在线未绑定设备
2. 为设备生成 6 位动态绑定码
3. 写入 `devices` 表
4. 向 `device/<DEVICE_UID>/bound` 发布 `event = dyn_bound_code` 的 JSON
5. 用户在云平台输入这 6 位码完成绑定

## 16. 结论

当前动态绑定码 MQTT 下发协议的关键点可以概括成一句话：

- **topic 负责指向设备**
- **event 负责区分语义**
- **dyn_bound_code 负责让用户完成绑定**

所以现阶段设备端最重要的是：

- 正确订阅 `device/<本机UID>/bound`
- 只处理 `event = dyn_bound_code`
- 把 6 位数字原样显示到屏幕
