# 11-墨水屏抖动bug修复-2026-04-24

## 1. 问题现象

当前项目在微雪 `spectra6 / E6` 全彩墨水屏上的渲染效果存在几个明显问题：

- 纯白区域不纯，容易出现黑点
- 纯黑区域不纯，容易出现白点
- 整体对比度偏低，画面发黑、发闷
- 多色照片显示偏苍白，颜色不够鲜艳

这些现象和“黑白点被错误混用、调色板不匹配、抖动算法选错”高度一致。

## 2. 根因定位

本次对比了两份实现：

- 项目当前实现：`backend/api/imgprocess/imgprocess.php`
- 官方参考实现：`notes/extern_files/1-ConverTo6c_bmp-7.3/convert.py`

### 2.1 当前项目算法与官方算法不一致

项目当前代码实际使用的是：

- `7 色` 调色板
- `Bayer Ordered Dithering` 有序抖动
- 额外引入了一个 `橙色 [255,165,0]`

而官方脚本实际使用的是：

- `6 色固定调色板`
- `Floyd-Steinberg` 误差扩散抖动
- 官方 6 色为：`黑、白、黄、红、蓝、绿`

官方脚本中的关键逻辑如下：

```python
pal_image.putpalette(
    (0,0,0, 255,255,255, 255,255,0, 255,0,0, 0,0,0, 0,0,255, 0,255,0)
    + (0,0,0)*249
)

quantized_image = resized_image.quantize(
    dither=display_dither,
    palette=pal_image
).convert('RGB')
```

虽然调色板里有一个重复黑色槽位，但最终有效颜色集合仍然只有 6 种，不包含橙色。

### 2.2 为什么会出现“白里有黑点、黑里有白点”

有序抖动本质是人为给像素叠加空间阈值矩阵，再去找最近颜色。

对于纯白、纯黑这种本来就应该稳定落到调色板极值的区域：

- Bayer 阈值会把局部像素人为推亮或压暗
- 再做最近色匹配时，就可能被推向错误颜色
- 最终表现为白底冒黑点、黑底冒白点

这不是硬件本身的问题，而是当前服务端抖动策略不适合 `spectra6 / E6` 官方 6 色量化链路。

### 2.3 为什么会“整体发黑、彩色发灰”

有两个叠加原因：

1. 当前实现不是官方 6 色误差扩散，而是 4x4 Bayer 有序抖动，容易让中间调出现规则性脏点。
2. 当前调色板多引入了 `橙色`，但硬件目标实际是官方 6 色，导致颜色映射方向偏离。

结果就是：

- 暗部更容易被压得更脏
- 中间调不够通透
- 高饱和照片被错误压缩到不理想的颜色组合里，看起来会更“苍白”

## 3. 本次修复内容

已修改文件：`backend/api/imgprocess/imgprocess.php`

### 3.1 把实际渲染入口切换为 Spectra 6 的 6 色误差扩散

原先：

- `generate_bmp()` 中调用 `apply_ordered_dither()`

现在：

- `generate_bmp()` 中改为调用 `apply_spectra6_dither()`

### 3.2 新增官方 6 色调色板

新增 `get_spectra6_palette()`，固定返回以下 6 色：

```php
[
    [0, 0, 0],
    [255, 255, 255],
    [255, 255, 0],
    [255, 0, 0],
    [0, 0, 255],
    [0, 255, 0]
]
```

对应：

- 黑
- 白
- 黄
- 红
- 蓝
- 绿

### 3.3 新增 Floyd-Steinberg 误差扩散实现

新增了这些辅助函数：

- `apply_spectra6_dither()`
- `create_error_row()`
- `clamp_channel()`
- `diffuse_error()`

误差扩散权重使用标准 Floyd-Steinberg：

- 右：`7/16`
- 左下：`3/16`
- 下：`5/16`
- 右下：`1/16`

这样做的直接收益是：

- 纯黑纯白不会再被阈值矩阵强行打散
- 中间调的过渡更自然
- 整体对比度和色彩识别更接近官方效果

### 3.4 更新缓存签名，避免旧错误 BMP 被复用

之前缓存签名只和渲染参数有关，不和算法版本绑定。

这会带来一个风险：

- 即使代码修好了，也可能继续命中旧缓存
- 设备看到的仍然是旧的错误 BMP

本次已把缓存签名改成包含算法版本：

```php
[
    'algorithm' => 'spectra6_fs_v1',
    'options' => $normalizedOptions
]
```

这样修复上线后，会自动生成新的 BMP 缓存，不会误用旧结果。

## 4. 预期修复效果

修复后预期会有这些变化：

- 白底更干净，黑点显著减少
- 黑底更稳定，白点显著减少
- 整体层次更清楚，不再明显发闷
- 彩色照片会比之前更接近官方转换效果
- 不再使用不属于目标 6 色集合的橙色

## 6. 建议的回归验证

建议在真实运行环境用这几类图片各测一张：

1. 纯白背景 + 黑字图
2. 纯黑背景 + 白字图
3. 高饱和照片（红黄蓝绿都明显）
4. 人像照片
5. 大面积浅色天空/墙面照片

重点观察：

- 白底是否还有随机黑点
- 黑底是否还有随机白点
- 黄色、红色、蓝色、绿色是否比之前更干净
- 整体是否比旧版本更亮、更通透

## 7. 结论

这次 bug 的核心不是 BMP 格式本身，而是“抖动算法和目标屏幕调色板不匹配”。

当前项目原先使用的 `7 色 Bayer 有序抖动` 不适合微雪 `spectra6 / E6` 官方 6 色渲染链路。  
本次已切换为与官方脚本一致方向的 `6 色固定调色板 + Floyd-Steinberg 误差扩散`，并同步处理了缓存版本问题。

如果后续还要继续优化，优先级建议是：

1. 在真实设备上做 A/B 对比截图
2. 如仍觉得“偏灰”，再考虑是否增加轻量级预处理（如亮度/对比度微调）
3. 保持调色板不扩色，避免再次偏离官方 6 色链路
