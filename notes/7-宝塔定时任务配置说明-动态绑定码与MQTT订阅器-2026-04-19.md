# 宝塔定时任务配置说明：动态绑定码与 MQTT 订阅器

**文档日期**：2026-04-19  
**适用项目**：SmartFrameCloud  
**适用环境**：宝塔面板 + PHP 8.1 + MQTT Broker

## 1. 文档目的

本文档用于指导在宝塔面板中配置两个与智能相册设备绑定流程相关的定时任务：

1. `MQTT 订阅器保活任务`
2. `动态绑定码发布任务`

这样做的目的是让设备上行注册监听与动态绑定码下发逻辑在服务器重启、进程退出或任务轮转时都能稳定运行。

## 2. 两个定时任务分别做什么

## 2.1 MQTT 订阅器保活任务

对应脚本：

```text
/www/wwwroot/47.108.232.40_2026/cloud/backend/mqtt/subscribe_bound.php
```

作用：

- 常驻订阅 MQTT 主题 `device/+/bound`
- 接收设备首次注册上行消息
- 将设备写入或更新到 `devices` 表

注意：

- 这个脚本是**常驻进程**
- 不能每分钟直接重复启动一次
- 正确做法是：每分钟检查它是否还活着，不活才重启

## 2.2 动态绑定码发布任务

对应脚本：

```text
/www/wwwroot/47.108.232.40_2026/cloud/backend/mqtt/publish_bind_codes.php
```

作用：

- 扫描所有“在线且未绑定”的设备
- 统一轮转动态绑定码
- 通过 MQTT 将 6 位动态绑定码下发到设备

注意：

- 这个脚本是**短任务**
- 跑完就退出
- 适合每分钟执行一次

## 4. 任务一：MQTT 订阅器保活

## 4.1 为什么不能直接每分钟运行 subscriber

错误示例：

```bash
/www/server/php/81/bin/php /www/wwwroot/47.108.232.40_2026/cloud/backend/mqtt/subscribe_bound.php
```

如果你把它直接放进“每分钟执行一次”的定时任务里，会出现：

- 每分钟都会新起一个 subscriber
- 服务器里会跑多个重复监听进程
- 同一条 MQTT 消息可能被多个进程重复处理

因此不能这么做。

## 4.2 正确方式：保活检查

正确思路是：

- 每分钟检查一次 subscriber 是否存在
- 如果存在，不做任何事
- 如果不存在，就后台启动一个

## 4.3 宝塔配置项

任务类型：

- `Shell脚本`

任务名称：

- `MQTT订阅器保活`

执行周期：

- 选择 `N分钟`
- 填 `1`

执行用户：

- `root`

开启进程锁：

- 建议勾选

## 4.4 脚本内容

把下面这整行填入“脚本内容”：

```bash
pgrep -f "php /www/wwwroot/47.108.232.40_2026/cloud/backend/mqtt/subscribe_bound.php" > /dev/null || nohup /www/server/php/81/bin/php /www/wwwroot/47.108.232.40_2026/cloud/backend/mqtt/subscribe_bound.php >> /www/wwwroot/47.108.232.40_2026/cloud/backend/mqtt/subscribe_bound.log 2>&1 &
```

说明：

- `pgrep -f`：检查 subscriber 是否已在运行
- `> /dev/null`：不输出检查结果
- `||`：如果前面没找到进程，就执行后面的启动命令
- `nohup ... &`：后台启动进程，SSH 断开后仍继续运行
- `>> ... 2>&1`：把标准输出和错误输出都写入日志文件

## 4.5 这个任务的最终效果

效果是：

- 服务器重启后，1 分钟内会自动拉起 subscriber
- subscriber 异常退出后，1 分钟内会自动恢复
- 不会重复起多个相同进程

- 填 `1`

执行用户：

- `root`

## 5.3 脚本内容

把下面这整行填入“脚本内容”：

```bash
/www/server/php/81/bin/php /www/wwwroot/47.108.232.40_2026/cloud/backend/mqtt/publish_bind_codes.php >> /www/wwwroot/47.108.232.40_2026/cloud/backend/mqtt/publish_bind_codes.log 2>&1
```

说明：

## 8. 常见错误

## 8.1 把 crontab 表达式写进脚本内容

错误示例：

```bash
* * * * * /www/server/php/81/bin/php ...
```

在宝塔里这是错的。

原因：

- 宝塔已经单独让你选择“执行周期”
- 脚本内容里只需要写**真正执行的命令**

## 8.2 用 subscriber 直接做每分钟任务

错误示例：

```bash
/www/server/php/81/bin/php /www/wwwroot/47.108.232.40_2026/cloud/backend/mqtt/subscribe_bound.php
```

然后设成“每分钟执行一次”。

这会导致：

- 多开进程
- 重复消费
- 运维混乱

## 8.3 PHP CLI 路径写错

如果脚本执行时报：

- `No such file or directory`
- `command not found`

说明 PHP CLI 路径不对。

先用下面命令确认：

```bash
/www/server/php/81/bin/php -v
```

## 9. 最终建议

本项目当前推荐采用以下组合：

### 常驻类

- `subscribe_bound.php`
- 用宝塔定时任务做“保活”

### 轮询类

- `publish_bind_codes.php`
- 用宝塔定时任务每分钟执行一次

这样的优点是：

- 配置简单
- 运维可控
- 服务器重启后可自动恢复
- 不需要额外安装 Supervisor

## 10. 结论

如果你想在宝塔里统一管理：

- `subscribe_bound.php` 不能当成普通每分钟脚本直接执行
- 正确做法是“保活检查 + 后台启动”
- `publish_bind_codes.php` 则可以作为普通分钟任务执行

这两个任务配好之后，就能形成一套较稳定的设备注册监听与动态绑定码轮转机制。
