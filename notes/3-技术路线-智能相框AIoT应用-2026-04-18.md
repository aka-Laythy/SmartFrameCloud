# 智能相框 AIoT 应用技术路线

**文档版本**：2026-04-19  
**文件名保留**：`3-技术路线-智能相框AIoT应用-2026-04-18.md`  
**适用项目**：`SmartFrameCloud`  
**文档定位**：按当前代码实现整理的技术路线说明，不再按“预期方案”描述。

## 1. 项目定位

`SmartFrameCloud` 是一个面向智能相框/智能相册设备的轻量级 AIoT 云平台，当前采用单体 Web 应用形态实现：

- 管理端：原生 HTML + CSS + JavaScript
- 服务端：PHP 8.1
- 数据存储：MySQL
- 设备消息通道：MQTT
- 图片分发通道：HTTP
- 文件存储：本地磁盘 `uploads/` 与 `cache/`

当前版本已经形成两条核心业务闭环：

1. 用户上传图片，平台处理为设备可显示的 BMP，再通过 MQTT 下发图片地址。
2. 设备上线后向云端注册，云端为未绑定设备轮转动态绑定码，用户通过 6 位码完成绑定。

这意味着项目重点已经从“静态设备 UID 直绑”转向“设备上线注册 + 动态绑定码 + Web 端认领”的实际可运行路线。

## 2. 总体技术路线

### 2.1 架构思路

项目采用“浏览器管理端 + PHP API + MySQL + MQTT Worker + 本地文件存储”的组合式单体架构：

1. 浏览器管理端负责用户登录、相册管理、设备绑定、图片下发操作。
2. PHP API 负责认证、相册、图片、设备等业务接口。
3. MySQL 保存用户、相册、图片、设备、下发日志、动态绑定码状态。
4. `backend/mqtt/subscribe_bound.php` 常驻订阅设备注册上行消息。
5. `backend/mqtt/publish_bind_codes.php` 按批扫描未绑定在线设备，并下发动态绑定码。
6. 设备通过 MQTT 收到控制消息，通过 HTTP 拉取 BMP 资源。

### 2.2 当前系统组成

```text
浏览器管理端
  ├─ 登录 / 注册
  ├─ 控制台
  ├─ 相册管理
  └─ 设备管理
        |
        v
PHP API
  ├─ auth
  ├─ user
  ├─ albums
  ├─ upload
  ├─ imgprocess
  └─ devices
        |
        +--> MySQL
        +--> uploads/（原图）
        +--> cache/（BMP 缓存）
        |
        +--> MQTT 发布
        |
        +--> MQTT Worker
              ├─ subscribe_bound.php
              └─ publish_bind_codes.php
        |
        v
MQTT Broker
        |
        v
智能相框设备
  ├─ 上行：device/<DEVICE_UID>/bound
  └─ 下行：device/<DEVICE_UID>/bound、device/<DEVICE_UID>/image
```

### 2.3 技术选型

#### 前端

- 原生多页面结构：`index.html`、`register.html`、`dashboard.html`、`albums.html`、`devices.html`
- 统一样式：`css/style.css`
- 原生 JavaScript + `fetch` + `FormData`
- 在设备页内完成绑定、图片选择、BMP 预览和渲染参数调整

#### 后端

- PHP 8.1
- PDO 访问 MySQL
- Session 维持登录态
- GD 负责验证码生成与 BMP 图像处理
- 自实现 MQTT v3.1.1 连接、订阅、发布基础逻辑

#### 存储

- `uploads/`：原始图片
- `cache/`：BMP 缓存
- MySQL：业务状态与审计记录

## 3. 前端实现路线

### 3.1 页面职责

- `index.html`：登录页，依赖验证码接口
- `register.html`：注册页，注册后自动创建默认相册
- `dashboard.html`：控制台，展示用户和资源统计
- `albums.html`：相册创建、图片浏览、图片上传、封面设置
- `devices.html`：设备绑定、设备列表、图片下发、BMP 预览调整

### 3.2 当前前端交互特点

前端仍然是“页面驱动 + 接口驱动”的轻量实现，但 `devices.html` 已经明显承担更重的业务逻辑：

- 绑定设备时不再输入设备 UID，而是输入设备屏幕上的 6 位动态绑定码
- 下发图片前，前端允许选择：
  - 横屏 `800 x 480` 或竖屏 `480 x 800`
  - 是否顺时针旋转 `90°`
  - 兴趣区中心点 `crop_center_x / crop_center_y`
  - 裁剪缩放系数 `crop_zoom`
- 前端在本地 Canvas 中做“软预览”，但不会修改原图文件

这条路线保持了 KISS：不引入前端框架，也不提前做组件化重构，而是在单页内补足设备操作闭环。

## 4. 后端模块设计

### 4.1 公共基础层

`backend/config/database.php` 统一维护：

- MySQL 连接配置
- MQTT Broker 连接配置
- 站点地址 `APP_URL`
- 上传路径 `UPLOAD_PATH`
- 上传访问前缀 `UPLOAD_URL`
- 最大上传限制 `MAX_UPLOAD_SIZE`
- 会话初始化与时区

`backend/includes/functions.php` 当前承担的职责较多，属于项目的共享基础层：

- JSON 响应
- 登录鉴权
- 验证码生成与校验
- 设备 UID 与动态绑定码校验
- 图片上传
- MQTT 发布
- 动态绑定码下发
- 图片下发设备
- BMP 渲染参数规范化

### 4.2 认证与用户模块

接口目录：

- `backend/api/auth/`
- `backend/api/user/`

当前能力：

- 验证码生成
- 用户注册、登录、退出
- 会话登录态校验
- 控制台统计数据读取

### 4.3 相册与图片模块

接口目录：

- `backend/api/albums/`
- `backend/api/upload/`

当前能力：

- 相册创建、删除、列表查询
- 相册图片列表查询
- 相册封面设置
- 图片上传并入库

当前上传链路支持的图片类型：

- JPEG
- PNG
- GIF
- WebP

BMP 处理链路在 GD 能力允许时也会尝试处理以上格式，其中 WebP 依赖运行环境是否启用了 `imagecreatefromwebp()`。

### 4.4 图片处理与 BMP 生成模块

核心文件：

- `backend/api/imgprocess/imgprocess.php`

当前实现不是简单固定转图，而是“带渲染参数的 BMP 生成器”：

- 输入方式支持：
  - `image_id`
  - `file`
  - `source`
- 渲染参数支持：
  - `orientation`
  - `rotate`
  - `crop_center_x`
  - `crop_center_y`
  - `crop_zoom`
- 输出分辨率：
  - 横屏 `800 x 480`
  - 竖屏 `480 x 800`
- 调色方式：
  - 7 色调色板
  - 有序抖动
- 输出格式：
  - 24 位 BMP V3

当前缓存策略也已经升级：

- BMP 文件名附带渲染参数签名
- 同一原图在不同方向、缩放、裁剪参数下可生成多份独立缓存
- 旧缓存会根据源文件更新时间和文件合法性做失效判断

### 4.5 设备管理模块

接口目录：

- `backend/api/devices/list.php`
- `backend/api/devices/bind.php`
- `backend/api/devices/unbind.php`
- `backend/api/devices/send-image.php`

当前设备管理路线的核心变化是：

- 旧路线：用户输入设备 UID 直接绑定
- 新路线：设备先上线注册，平台轮转动态绑定码，用户输入动态绑定码认领设备

`bind.php` 当前按以下规则工作：

1. 接收 6 位数字动态绑定码 `dyn_bound_code`
2. 查找 `devices` 表中“未绑定且未过期”的设备
3. 更新 `user_id`、`name`、`description`、`bound_at`
4. 清空该设备上的动态绑定码相关字段

`unbind.php` 会清空：

- 用户归属
- 设备名称与描述
- 当前图片
- 绑定时间
- 动态绑定码相关字段

### 4.6 设备图片下发模块

核心函数：

- `backend/includes/functions.php` 中的 `sendImageToDevice()`

当前下发流程：

1. 校验设备归属与图片归属
2. 从 `uploads/` 中找到原图
3. 按当前渲染参数生成 BMP
4. 通过 `imget.php?file=...` 暴露 BMP 下载地址
5. 向 `device/<DEVICE_UID>/image` 发布 MQTT 消息
6. 记录 `device_image_logs`
7. 成功时更新 `devices.current_image_id`

当前图片下发消息包含：

- `image_id`
- `image_url`
- `original_name`
- `width`
- `height`
- `orientation`
- `rotate`
- `timestamp`

相比旧版技术路线，当前下发协议已经不只是“固定分辨率图片地址通知”，而是包含了设备渲染方向语义。

### 4.7 MQTT Worker 模块

这是当前文档最需要更新的部分。

#### `backend/mqtt/subscribe_bound.php`

职责：

- 常驻连接 MQTT Broker
- 订阅 `device/+/bound`
- 接收设备首次注册/上线消息
- 将设备写入或更新到 `devices`

当前只接受如下语义的上行消息：

```json
{
  "event": "reg_new_device",
  "device_uid": "AC12EF34AB56CD78",
  "timestamp": 1776530000
}
```

处理结果：

- 不存在则插入设备
- 已存在则刷新：
  - `status = 1`
  - `last_online_at = NOW()`

它同时会忽略云端下发到同一 topic 的动态绑定码消息，即按 `event = dyn_bound_code` 区分上下行语义。

#### `backend/mqtt/publish_bind_codes.php`

职责：

- 周期性扫描“在线且未绑定”的设备
- 为设备生成 6 位动态绑定码
- 写入：
  - `dyn_bound_code`
  - `dyn_bound_code_issued_at`
  - `dyn_bound_code_expires_at`
- 向 `device/<DEVICE_UID>/bound` 下发动态绑定码消息

当前默认策略：

- 绑定码有效期：300 秒
- 提前刷新窗口：60 秒
- 单轮最多处理：100 台设备
- 单设备发布间隔：50 ms

当前下发 payload：

```json
{
  "event": "dyn_bound_code",
  "device_uid": "AC12EF34AB56CD78",
  "dyn_bound_code": "483726",
  "expires_in": 300,
  "timestamp": 1776530000
}
```

### 4.8 BMP 文件访问模块

入口文件：

- `imget.php`

职责：

- 从 `cache/` 中读取 BMP
- 限制只能访问 `.bmp`
- 进行路径清洗，避免目录遍历
- 返回 `Content-Type: image/bmp`

这条链路使设备只需具备 HTTP 下载能力，无需直接访问后台文件系统。

## 5. 核心业务闭环

### 5.1 设备绑定闭环

当前绑定流程如下：

1. 设备连上网络与 MQTT Broker。
2. 设备向 `device/<DEVICE_UID>/bound` 上报 `event = reg_new_device`。
3. `subscribe_bound.php` 将设备写入数据库，并标记为在线。
4. `publish_bind_codes.php` 为未绑定在线设备生成 6 位动态绑定码并下发到同一 topic。
5. 用户在 `devices.html` 中输入动态绑定码，`bind.php` 完成设备认领。

这是当前项目最核心的 AIoT 联网闭环，已经替代旧文档中的“直接输入设备 UID 绑定”方案。

### 5.2 图片下发闭环

1. 用户上传图片到相册。
2. 用户在设备页中选择目标设备和目标图片。
3. 用户在前端调整方向、旋转、兴趣区和缩放。
4. `send-image.php` 调用 `sendImageToDevice()` 生成对应 BMP。
5. 后端向 `device/<DEVICE_UID>/image` 发布图片消息。
6. 设备再通过 `imget.php?file=...` 拉取 BMP 并显示。

## 8. 部署路线

### 8.1 基础环境

当前实现依赖：

- Nginx
- PHP 8.1
- MySQL 5.7+
- EMQX 或兼容 MQTT Broker
- PHP GD 扩展

### 8.2 运行模式

项目不是“只有网站服务”就能完整运行，当前至少包含三类运行单元：

1. Web 服务：处理页面与 PHP API 请求
2. MQTT 常驻订阅器：`subscribe_bound.php`
3. 动态绑定码发布任务：`publish_bind_codes.php`

### 8.3 当前推荐运维方式

从现有文档与代码看，当前部署路线是：

- `subscribe_bound.php` 作为常驻进程运行
- 用定时任务做保活检查
- `publish_bind_codes.php` 作为周期任务按分钟执行

这比把所有 MQTT 行为都放在页面请求里更稳，也符合当前项目规模下的 KISS 设计。

### 9.1 已实现能力

- BMP 缓存复用

- 设备解绑
- 设备图片 MQTT 下发
- BMP 安全读取

### 9.2 当前边界与技术债

当前仍有以下边界需要明确记录：

- `device/<DEVICE_UID>/bound` 同时承担设备上行注册与云端下行绑定码两种语义，目前依赖 `event` 字段区分
- 尚未实现独立的设备心跳 topic 与离线回收逻辑，`status = 2` 目前没有形成自动闭环
- 相册侧仍没有单张图片删除接口
- 相册页还未直接提供“从相册联动下发到设备”的入口
- 图片下发地址仍写死公网主机 `http://47.108.232.40:2026/cloud/...`，部署迁移时需要同步调整
- 配置文件中仍存在环境硬编码项，尚未完成环境变量化
- 当前 MQTT 下发实现偏轻量，尚未构建完整的企业级可靠投递机制

## 10. 下一阶段演进建议

### 10.1 设备在线状态闭环

建议补齐独立状态 topic，例如：

- `device/<DEVICE_UID>/status`
- `device/<DEVICE_UID>/msg`

并增加：

- 在线心跳写回
- 超时离线判定
- 设备最后活跃时间可视化

### 10.2 绑定协议解耦

建议中期把“注册上行”和“绑定码下行”拆到不同 topic，减少同 topic 复用带来的歧义。

### 10.3 图片链路增强

建议补齐：

- 单张图片删除
- 相册页直达设备下发
- 异步 BMP 预生成
- 更稳定的外部访问地址配置

### 10.4 架构演进

当设备量和用户量上升后，可进一步考虑：

- 将 MQTT Worker 独立成专门服务
- 将图片存储迁移到对象存储
- 引入异步任务队列处理图片渲染
- 视前端复杂度再决定是否迁移到组件化框架

## 11. 当前结论

截至当前代码版本，`SmartFrameCloud` 已不再只是“相册上传 + 图片下发”的原型，而是已经具备以下 AIoT 主干能力：

- 设备可以主动上线注册
- 云端可以为未绑定设备下发动态绑定码
- 用户可以通过动态绑定码认领设备
- 用户可以按设备朝向与兴趣区生成 BMP 并下发
- 设备可以通过 MQTT + HTTP 联合完成图片展示

从技术路线角度看，项目当前最准确的定位是：

**一个基于 PHP 单体应用、通过 MQTT Worker 补足设备侧闭环的智能相框云平台。**

后续最值得投入的方向不是继续堆页面，而是优先补齐设备状态闭环、配置参数化和消息协议分层。
